# The generate-invoke.ts Script

## Location

Source: `packages/cli/scripts/generate-invoke.ts` (this repo, published as `@decocms/cli`).
Site usage: `node_modules/@decocms/cli/scripts/generate-invoke.ts`.

This moved from the old single-package `@decocms/start/scripts/generate-invoke.ts` when the framework split into `runtime`/`admin`/`cli`/`tanstack`/`next` — see the root `README.md`. `@decocms/cli`'s own `package.json` also declares a `"./generate-invoke"` subpath export pointing at the same script, but the script is a standalone `tsx` entry point, not a library import — invoke it by file path as shown below, not via the package export.

## What It Does

1. Finds `@decocms/apps/vtex/invoke.ts` (checks `node_modules/@decocms/apps` and `../apps-start`) — `@decocms/apps` is a separate, independently published package/repo, not part of this monorepo
2. Parses it with `ts-morph` to extract action definitions
3. For each action, extracts: name, import source, imported function, input type, return type, unwrap flag, call body
4. Generates `src/server/invoke.gen.ts` with:
   - Top-level `createServerFn().inputValidator().handler()` declarations (one per action)
   - A `forwardResponseCookies()` helper, called at the end of every handler
   - A `vtexActions` export (for merging into a site-local `invoke.ts`) and a default `invoke` re-export

## Usage

```bash
# From site root
npx tsx node_modules/@decocms/cli/scripts/generate-invoke.ts

# With custom output
npx tsx node_modules/@decocms/cli/scripts/generate-invoke.ts --out-file src/invoke.gen.ts

# With custom apps location
npx tsx node_modules/@decocms/cli/scripts/generate-invoke.ts --apps-dir ../apps-start
```

## CLI Options

| Option | Default | Description |
|--------|---------|-------------|
| `--out-file` | `src/server/invoke.gen.ts` | Output file path |
| `--apps-dir` | Auto-detected (`node_modules/@decocms/apps`, then `../apps-start`) | Path to `@decocms/apps` root |

## How It Parses invoke.ts

The script navigates the AST:

```
invoke (VariableDeclaration)
  └── { vtex: { actions: { ... } } } as const  (AsExpression → ObjectLiteral)
        └── vtex (PropertyAssignment)
              └── actions (PropertyAssignment)
                    └── getOrCreateCart: createInvokeFn(...) as (...)
                    └── addItemsToCart: createInvokeFn(...) as (...)
                    └── ...
```

For each property in `actions`:
1. Strips the `as Type` assertion to get the `createInvokeFn(...)` call
2. Extracts the arrow function parameter type → `inputType`
3. Extracts the arrow function body **verbatim** as `callBody` — e.g. `getOrCreateCart(data)` for a direct pass-through, or `createSession({ data })` for a wrapper that adapts the payload shape. This is emitted as-is in the generated handler (`const result = await ${callBody};`), so adapting wrappers keep working; hard-coding `${importedFn}(data)` for every action was tried and reverted because it silently dropped the adaptation (see the generator's inline comment and `generate-invoke.test.ts`'s "preserves adapting wrappers verbatim" case)
4. Checks for `{ unwrap: true }` in the second argument
5. Resolves which imported function is called (matches against import map)
6. Extracts the return type from the outermost `as` assertion

The parser does **not** execute `createInvokeFn` — it only reads source text via `ts-morph`. Whether `createInvokeFn` itself resolves to a real export at runtime is a concern for `@decocms/apps`'s own build, not for this generator.

## Generated Output Structure

This is the actual current shape (condensed from `packages/cli/scripts/generate-invoke.ts`; read the script for the full comment block explaining each design decision):

```typescript
// invoke.gen.ts
// Auto-generated by @decocms/cli/scripts/generate-invoke.ts
// Do not edit manually. Re-run the generator to update.

import { createServerFn } from "@tanstack/react-start";
import { addItemsToCart, ... } from "@decocms/apps/vtex/actions/checkout";
import type { OrderForm } from "@decocms/apps/vtex/types";
import {
  getResponseHeaders,
  setResponseHeader,
} from "@tanstack/react-start/server";
import { RequestContext } from "@decocms/runtime/sdk/requestContext";

function unwrapResult<T>(result: unknown): T {
  if (result && typeof result === "object" && "data" in result) {
    return (result as { data: T }).data;
  }
  return result as T;
}

// Forwards Set-Cookie headers captured in RequestContext.responseHeaders
// (by vtexFetchWithCookies) into TanStack Start's HTTP response. Uses
// Headers.getSetCookie() — not .entries() — because .entries() collapses
// multiple Set-Cookie values into one comma-joined string, which browsers
// silently discard. No-op when nothing was captured, so every handler can
// call it unconditionally.
function forwardResponseCookies(): void {
  const ctx = RequestContext.current;
  if (!ctx) return;
  const captured =
    typeof ctx.responseHeaders.getSetCookie === "function"
      ? ctx.responseHeaders.getSetCookie()
      : [];
  if (captured.length === 0) return;
  const existing =
    typeof getResponseHeaders().getSetCookie === "function"
      ? getResponseHeaders().getSetCookie()
      : [];
  setResponseHeader("set-cookie", [...existing, ...captured]);
}

// ---------------------------------------------------------------------------
// Top-level server function declarations — compiler can transform these
// ---------------------------------------------------------------------------

const $addItemsToCart = createServerFn({ method: "POST" })
  .inputValidator((data: { orderFormId: string; orderItems: CartItem[] }) => data)
  .handler(async ({ data }): Promise<any> => {
    const result = await addItemsToCart(data.orderFormId, data.orderItems);
    forwardResponseCookies();
    return unwrapResult(result);   // only when the action declared { unwrap: true }
  });

// ---------------------------------------------------------------------------
// Typed VTEX actions map — merge with site-specific actions in your invoke.ts
// ---------------------------------------------------------------------------

export const vtexActions = {
  addItemsToCart: $addItemsToCart as unknown as (ctx: { data: { orderFormId: string; orderItems: CartItem[] } }) => Promise<OrderForm>,
} as const;

export type { OrderForm } from "@decocms/apps/vtex/types";

// ---------------------------------------------------------------------------
// Default invoke object — import this if you don't need site extensions
// ---------------------------------------------------------------------------

export const invoke = {
  vtex: {
    actions: vtexActions,
  },
} as const;
```

Two things every earlier version of this doc was missing, both load-bearing:

- **`.inputValidator((data: InputType) => data)`** before `.handler()`. Without it, TanStack Start types the handler's `data` as `undefined` and every field access is a type error. `troubleshooting.md` and `architecture.md` already assumed this; this doc now matches.
- **`forwardResponseCookies()`**, called after the `await` but before `return`, in *every* generated handler (unconditionally — it's a cheap no-op for read-only actions). This is the fix for the "cart forgets items" bug documented in `troubleshooting.md`. `generate-invoke.test.ts` locks both the call-count (one per handler) and the ordering (after the await).

Handlers whose action wasn't parseable are emitted as throwing stubs (`// TODO: could not auto-generate ${name} — add manually`) rather than silently omitted — check the generator's console warnings after running it.

## Adding New Actions

1. Add the pure function to `@decocms/apps/vtex/actions/{module}.ts` (in the separate `@decocms/apps` repo)
2. Add it to `@decocms/apps/vtex/invoke.ts` using `createInvokeFn`
3. Re-run `npm run generate:invoke` in the site
4. The new action is automatically available in `vtexActions.*` / `invoke.vtex.actions.*` (if your site's `src/server/invoke.ts` re-exports `vtexActions` — see `architecture.md`'s "Layer 3.5")

## unwrapResult

VTEX checkout functions return `VtexFetchResult<T>` which wraps `{ data: T, setCookies: string[] }`. The `unwrap: true` flag causes the generated handler to extract `.data` before returning to the client:

```typescript
// Without unwrap:
const result = await simulateCart(items, postalCode);
return result;  // returns raw simulation response

// With unwrap:
const result = await getOrCreateCart(orderFormId);
return unwrapResult(result);  // returns OrderForm, not { data: OrderForm, setCookies }
```

## Site-Local Composition (`invoke.ts`)

`generate-invoke.ts` only emits the canonical VTEX action set. Sites needing custom/site-specific server functions write those by hand in `src/server/invoke.ts`, importing and spreading `vtexActions` from the generated file. That hand-written layer follows the same `.inputValidator()` + `Promise<any>` pattern documented in `.agents/skills/deco-to-tanstack-migration/references/server-functions/README.md` — but must additionally call `forwardResponseCookies()` for any action touching VTEX cookies, matching what the generator does. See `architecture.md`'s "Layer 3.5" for the full example.

## Integration with Build Pipeline

The script fits into the existing build pipeline alongside other generators:

```json
{
  "scripts": {
    "generate:blocks": "tsx node_modules/@decocms/cli/scripts/generate-blocks.ts",
    "generate:invoke": "tsx node_modules/@decocms/cli/scripts/generate-invoke.ts",
    "generate:schema": "tsx node_modules/@decocms/cli/scripts/generate-schema.ts",
    "build": "npm run generate:blocks && npm run generate:invoke && npm run generate:schema && tsr generate && vite build"
  }
}
```

Order matters: `generate:invoke` should run before `generate:schema` (schema needs to see all imports), but after `generate:blocks` (blocks don't depend on invoke).
